# Web Chat Viva Notes (Full JS + Firebase Explanation)

## 1) Project Structure

- `index.html` → page structure (header, chat area, composer, modals)
- `styles.css` → all UI styling (light/dark themes, layout, buttons, modals)
- `script.js` → all behavior (Firebase, auth, chat logic, admin features)

This is called **separation of concerns**.

---

## 2) How Firebase is Connected

In `index.html`, Firebase libraries are loaded first:

- `firebase-app.js`
- `firebase-firestore.js`

Then `script.js` runs:

1. `firebase.initializeApp(firebaseConfig)`  
   Connects the app to Firebase project `web-chat-93bab`.
2. `const db = firebase.firestore()`  
   Creates Firestore DB instance for read/write/query/listen operations.

So every `db.collection(...)` line in JS is using this connected Firestore instance.

---

## 3) Firestore Database Design

## `users` collection

- Document ID: normalized username (lowercase), e.g. `yusuf`
- Fields:
  - `username` (display name)
  - `passwordHash` (SHA-256 hash, not plain password)
  - `createdAt`, `updatedAt`

Why unique username works:  
Firestore doc IDs are unique. Since doc ID is normalized username, duplicate usernames are prevented.

## `messages` collection

- Document ID: auto-generated by Firestore
- Fields:
  - `name` (sender display name)
  - `userId` (normalized username, used for ownership)
  - `message` (text content)
  - `time` (timestamp)

---

## 4) App State in JavaScript

Common state variables:

- `currentUser` → logged-in user info (`id`, `name`, `isAdmin`) or `null`
- `allMessages` → local message cache from Firestore listener
- Busy flags (`isAuthBusy`, `isUsersBusy`, `isSending`) → prevent double-click duplicate actions

## 5) Deep Function-by-Function Explanation

## A) Theme / Toggle Functions

### `setTheme(theme)`

- Sets `data-theme` on `<body>` (`light` or `dark`)
- Saves theme in `localStorage`
- Updates toggle button text/icon

### `toggleTheme()`

- Reads current theme
- Switches to the opposite value
- Calls `setTheme(newTheme)`

---

## B) Auth Utility Functions

### `normalizeUsername(username)`

- Trims spaces
- Converts to lowercase
- Used as stable account ID

### `getAuthInput()`

- Reads username/password from modal inputs
- Validates username format and password length
- Returns cleaned values for auth functions

### `hashPassword(password)`

- Uses browser crypto (`SHA-256`)
- Converts password to hash string
- Hash is stored and compared (never plain password)

### `isAdminCredentials(userId, password)`

- Returns `true` only for admin login rule (`admin` + `admin`)
- Used to grant admin-only actions

---

## C) Account Creation and Sign-In Functions

### `createNewAccount()`

Purpose: register first-time users.

Steps:

1. Read + validate inputs
2. Check if `users/{userId}` already exists
3. If exists → show “username already taken”
4. If not:
   - hash password
   - store user document in Firestore
5. Call `completeLogin(...)`

### `signInUser()`

Purpose: login already registered users.

Steps:

1. Read + validate inputs
2. Load `users/{userId}` from Firestore
3. If missing → “account not found”
4. Hash entered password
5. Compare with stored `passwordHash`
6. If match → `completeLogin(...)`; else show error

### `completeLogin(userId, displayName, isAdmin)`

- Sets `currentUser`
- Updates top user label
- Enables composer
- Shows/hides admin controls
- Closes auth modal
- Re-renders messages for ownership-based buttons

### `logoutAndSwitchUser()`

- Clears session
- Disables sending
- Reopens auth modal
- Lets user sign in as another account

---

## D) Chat Functions

### `sendMsg()`

- Validates user is logged in
- Validates message text (not empty, within max length)
- Adds a document to `messages` collection:
  - `name`, `userId`, `message`, `time`
- Clears input and updates character count

### `deleteMsg(messageId, ownerId, ownerName)`

- Allows delete only if logged-in user owns that message
- Confirms action
- Deletes specific message doc from Firestore

### `clearAllMessages()`

- Admin-only feature
- Confirms action
- Deletes all message docs in batches (safe and scalable)

---

## E) Message Rendering Functions

### `createMessageElement(doc)`

- Builds DOM node for one message:
  - sender
  - time
  - message text
- Adds delete button only if current user is allowed

### `renderMessages(shouldScrollToBottom)`

- Applies search filter if present
- Handles empty state
- Appends message elements to chat container
- Updates message count text
- Optionally auto-scrolls to bottom

### `formatTime(timestamp)`

- Converts numeric timestamp to readable local time

### `updateMessageCount(visible, total)`

- Shows:
  - `X messages`, or
  - `X of Y shown` when searching

### `updateCharCount()`

- Shows live input character count (example: `120/500`)

---

## F) Registered Users Modal Functions

### `openUsersModal()` / `closeUsersModal()`

- Show/hide users list modal
- View is allowed for all logged-in users

### `loadRegisteredUsers()`

- Fetches all `users` docs
- Sorts and sends data for rendering

### `renderUsersList(users)`

- Displays usernames
- For non-admin: only view
- For admin: delete controls enabled (except admin account)

### `deleteUserAccount(userId, username)`

- Admin-only
- Confirms deletion
- Deletes user doc
- Also deletes that user’s messages

### `deleteMessagesByUserId(userId)`

- Queries messages where `userId == target`
- Deletes in batches repeatedly until none left

## 6) Real-Time Sync (Most Important Viva Point)

`onSnapshot` listener is used on `messages` collection.

Meaning:

- No page refresh required
- Any new message/delete by any user instantly updates everyone’s UI
- Listener updates local `allMessages`, then `renderMessages(...)` redraws chat

---

## 7) Firestore Queries Used

- `db.collection("users").doc(userId).get()` → read one user
- `db.collection("users").doc(userId).set({...})` → create/update user
- `db.collection("users").get()` → list all users
- `db.collection("users").doc(userId).delete()` → delete user
- `db.collection("messages").add({...})` → send message
- `db.collection("messages").doc(id).delete()` → delete one message
- `db.collection("messages").where("userId","==",x).limit(n).get()` → query user messages
- `db.batch()` + `batch.delete(...)` + `batch.commit()` → bulk delete
- `db.collection("messages").orderBy("time").onSnapshot(...)` → real-time updates

---

## 8) Startup Flow (What happens when page opens)

1. Firebase scripts load
2. `script.js` runs and initializes Firebase/Firestore
3. DOM element references are collected
4. Event listeners are attached (buttons, forms, search, theme)
5. Theme is loaded from `localStorage`
6. Message real-time listener starts (`onSnapshot`)
7. User session starts as logged out
8. Composer disabled until login
9. Auth modal opens

---

## 9) Admin Behavior Summary

- Admin identity: username `admin`, password `admin`
- Admin can:
  - clear all messages
  - delete user accounts
  - remove user messages
- All signed-in users can:
  - view registered users list
- Only admin can perform destructive user actions
